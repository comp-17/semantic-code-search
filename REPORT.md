## Track Declaration

**Track B — Code RAG (Short)**

This report covers a Retrieval-Augmented Generation (RAG) pipeline built over Python source code. The starter corpus is drawn from the CodeSearchNet dataset (Python split, 1,000 functions). Five author-written utility functions were added in Part 2. All embeddings use `sentence-transformers/all-MiniLM-L6-v2`. Generation is handled by the UTSA-hosted `llama-3.3-70b-instruct-awq` endpoint via an OpenAI-compatible API. The vector database is a persistent ChromaDB collection with cosine similarity as the distance metric and top-k = 4.

---

## Pipeline Summary

The pipeline consists of three modules:

- **`retriever.py`** — Exposes a single public function `retrieve(query, k, config_path)` that internally loads config, initializes the embedding model (`all-MiniLM-L6-v2`), connects to ChromaDB, and returns the top-k most similar code chunks with metadata (repo, path, func_name, source, similarity score).
- **`generator.py`** — Assembles a grounded context block from the retrieved chunks (formatted as `[N] repo: ..., path: ..., func: ...`) and calls the Llama 3.3 70B model with a strict system prompt instructing it to answer only from the provided context and cite every source as `repo/path::func_name`.
- **`run_part1.py` / `run_part2.py`** — End-to-end drivers that load and index the corpus, run all queries, and write results tables to Markdown files. Both scripts are idempotent — re-running skips indexing if the collection already contains the expected items.

## Reflection

**When did retrieval succeed, and when did it return topically similar but not actually useful chunks?**

Retrieval performed best when the query vocabulary closely matched the function's docstring terminology. For example, Q01 ("parse command line arguments") retrieved `Command.create_parser` with a score of 0.6947, and Q05 ("retry mechanism") retrieved the `retry` function with 0.6064 — both high-confidence, directly useful matches. Retrieval struggled on Q06 ("flatten a nested list") and Q09 ("extract all URLs using regular expressions"), where the top-4 chunks were topically adjacent but did not implement the exact pattern asked about — Q06 returned a list-formatting helper, a zip utility, and a flat stats collector, none of which demonstrate flattening. In these cases, the generator correctly declined to answer rather than hallucinate, which is the desired behavior. The pattern is clear: queries that use the same domain-specific verbs as the indexed docstrings retrieve well; queries that use more general or paraphrased language retrieve weaker results.

**What happened when you ran cross-corpus queries? Did the new items get retrieved when they should have?**

The new author-written functions were retrieved consistently for cross-corpus queries, appearing as the top-1 result in 4 out of 5 cases (C01, C02, C03, C05). This confirms that the local functions were correctly embedded and inserted into the shared collection, and that their docstrings were semantically aligned with the cross-corpus query phrasings. C04 ("rank items by a numeric score") was the exception — it retrieved a CodeSearchNet function (`render_value`) as top-1, with `top_k_similar` from the local set appearing only at rank 4. Despite this, the generator still cited `top_k_similar` correctly in its answer. All cross-corpus results had "Mixed" source sets, meaning both starter and local chunks appeared among the top-4, demonstrating that the single shared collection enables genuine cross-corpus retrieval without any special routing logic.

**What is one thing you would change about the pipeline if you had another day?**

The most impactful improvement would be adding a re-ranking step between retrieval and generation. Currently the generator receives the raw top-k results ranked by embedding cosine similarity alone, which can surface topically adjacent but semantically shallow matches (as seen in Q06 and C04). A cross-encoder re-ranker — such as `cross-encoder/ms-marco-MiniLM-L-6-v2` — would score each (query, chunk) pair jointly, producing a much more precise shortlist before the prompt is assembled. This would reduce the frequency of "insufficient context" answers without requiring a larger starter corpus or a different embedding model.
